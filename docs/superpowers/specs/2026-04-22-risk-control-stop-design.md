# 风控异常提示自动停机设计

## 背景

当前仓库已经具备两类“高风险页面”硬停能力：

- 登录阶段命中 `#serviceAbuseLandingTitle` 时，直接报错并中止登录流程
- Rewards 页命中 `#suspendedAccountHeader` 时，直接报错并中止当前账号流程

但这两处仍然是分散的、点状的判断。对于更常见的“异常行为”“搜索受限”“账号受限”“earning limit”类提示，项目目前没有统一检测与统一停机链。

用户已确认本阶段目标不是规避风控，而是在检测到高置信风险提示后，立即停止整个项目继续运行，避免把异常状态继续扩大。

## 目标

- 为“异常行为/账号受限/earning limit”类页面提示建立统一检测器
- 命中后，不只是当前账号失败，而是整个项目立即停止
- 尽量复用现有浏览器关闭、日志告警、webhook 通知和进程退出链
- 降低误报，不做全页面模糊扫词
- 保持改动集中、可测试、可回退

## 用户确认的边界

- 任何账号命中高置信异常提示时，整个项目停止，不让其他账号继续跑
- 采用“混合检测”而不是激进全局扫词
- 先做识别和停机链，不做自动截图、自动导出 HTML 等附加功能

## 范围

本阶段只处理“检测并停机”，不处理：

- 绕过或规避风控
- 自动申诉
- 自动截图归档
- 自动切换代理、账号、节奏
- 扩展新的 Rewards 活动执行能力

预计主要改动范围：

- `src/index.ts`
- `src/browser/BrowserUtils.ts` 或单独新增风险检测模块
- `src/constants.ts`
- `src/interface/Config.ts`
- `src/util/Validator.ts`
- 相关测试

## 方案对比

### 方案 A：继续维持现状，只保留已存在的两处硬停

优点：

- 风险最低
- 改动最小

缺点：

- 只能覆盖已知的锁号/暂停页面
- 对“异常行为”“earning limit”类提示没有统一保护

### 方案 B：统一风险检测器 + 全局停机链

优点：

- 覆盖登录后、dashboard、搜索和活动落地页等关键节点
- 改动集中，便于维护规则
- 可以把“普通失败”和“风控致命事件”分开处理

缺点：

- 需要新增专用错误类型和 cluster 停机分支
- 需要谨慎控制关键词，避免误报

### 方案 C：全局监听页面内容变化，随时全文扫词

优点：

- 理论覆盖最大

缺点：

- 最容易误报
- 最影响现有运行性能
- 实现噪音大，调试成本高

推荐采用方案 B。

## 核心设计

### 1. 统一风险检测器

新增统一检测器，例如：

- `src/browser/RiskControlDetector.ts`

职责：

- 接收 `page`、当前阶段、账号邮箱
- 检测是否出现高置信风险提示
- 命中时返回结构化结果，未命中返回 `null`

检测逻辑采用两层：

#### 第一层：明确选择器

保留并集中管理已知高风险选择器，例如：

- `#serviceAbuseLandingTitle`
- `#suspendedAccountHeader`

并预留后续补充其他明确 banner / heading 选择器的能力。

#### 第二层：高置信中英文固定文案

只检查小范围、高置信关键词，不做模糊分词或相似匹配。

候选示例：

- `unusual activity`
- `restricted`
- `limited`
- `earning limit`
- `not following the rules`
- `暂停`
- `受限`
- `异常行为`

规则要求：

- 关键词集中定义在一处
- 仅作为兜底，不单独扩大到模糊匹配
- 后续若发现误报，优先删词，不增加复杂 NLP 逻辑

### 2. 限定检测触发点

检测器只在关键节点执行，不做全流程高频轮询。

本阶段挂载点：

- 登录完成后
- dashboard 加载后
- 每轮搜索开始前
- 每轮搜索结束后
- 活动新标签页打开并完成落地后

这样做的目的是：

- 既覆盖高风险阶段
- 又避免到处插入全文扫描导致误判和维护成本上升

### 3. 引入专用致命错误类型

新增专用错误，例如：

- `RiskControlDetectedError`

其语义与普通流程错误不同：

- 普通错误：当前账号记失败，现有流程继续
- 风控异常：立即升级为全局致命事件

错误中建议携带：

- `accountEmail`
- `stage`
- `matchedSelector`
- `matchedText`
- `message`

### 4. 单进程停机链

当前单进程模式下，普通账号错误会在 `runTasks()` 中被记录后继续跑下一个账号。

本阶段需要让 `RiskControlDetectedError` 走特殊分支：

- 不在 `runTasks()` 中吞掉
- 直接向上抛到主运行链
- 复用现有：
  - `logger.alert(...)`
  - `flushAllWebhooks()`
  - `process.exit(1)`

这样能保证：

- 当前账号立即停止
- 后续账号不会继续跑
- 退出码明确表示异常终止

### 5. 多进程停机链

当前 cluster 模式下，某个 worker 失败不会立刻终止其他 worker。

本阶段需要新增专用 IPC 紧急停机信号，例如：

- `__riskControlStop`

流程如下：

1. worker 命中风险检测
2. worker 发送结构化停机消息给主进程
3. 主进程记录一次全局 `alert`
4. 主进程向所有仍存活的 worker 发送终止信号
5. 等待 worker 退出
6. flush webhook 并以非 0 退出

要求：

- 只要出现第一次命中，就进入“全局停止中”状态
- 避免多个 worker 同时重复触发，造成重复杀进程和重复告警

### 6. 浏览器与资源清理

不新增独立清理系统，尽量复用当前 `Main()` 里的 `finally`：

- 保存 cookies / session
- 关闭 context
- 关闭 root browser

这样即使是风控停机，也尽量做到“告警后干净退出”，不留下脏浏览器进程。

### 7. 配置开关

新增配置项：

```json
{
  "riskControlStop": {
    "enabled": true
  }
}
```

设计要求：

- 默认开启
- 只提供总开关，不在第一版暴露复杂词表配置
- 配置结构进入：
  - `src/interface/Config.ts`
  - `src/util/Validator.ts`
  - `src/config.example.json`

## 日志与告警

命中时应明确记录：

- 账号邮箱
- 触发阶段
- 命中的选择器或文案
- 当前动作：停止当前账号并触发全局停机

告警仍走现有 `logger.alert(...)` 通道，确保：

- 控制台可见
- 本地日志落盘
- Discord / ntfy / PushPlus 可收到

## 代码影响

预计涉及：

- `src/browser/RiskControlDetector.ts`
  - 新增统一检测逻辑
- `src/constants.ts`
  - 集中维护高置信选择器 / 文案
- `src/index.ts`
  - 引入专用错误类型
  - 单进程下向上抛出风控致命错误
  - cluster 下新增紧急停机 IPC 处理
- `src/functions/SearchManager.ts`
  - 在搜索前后插入检测点
- 若部分活动落地页已有统一导航辅助逻辑，可在对应浏览器活动执行器中插入检测点

## 测试策略

至少补三类测试：

### 1. 检测器识别测试

- 命中明确选择器时返回检测结果
- 命中高置信关键词时返回检测结果
- 普通页面文案不误判

### 2. 单进程停机测试

- 命中后抛出 `RiskControlDetectedError`
- `runTasks()` 不再继续处理后续账号

### 3. 多进程停机测试

- worker 上报风险停机 IPC
- 主进程只触发一次全局停机
- 其他 worker 被终止

## 风险与取舍

- 如果关键词过宽，会误报并提前终止整个任务
- 如果关键词过窄，会漏掉部分真实风险提示
- 第一版优先“低误报”，宁愿漏少量未知页面，也不做激进全文扫描

## 验收标准

- 单进程模式下，任意账号命中高置信风险提示时，整个运行立即停止
- 多进程模式下，任意 worker 命中高置信风险提示时，主进程终止所有 worker 并退出
- 命中事件通过 `alert` 通道可见
- 正常页面不出现明显误报
- 现有锁号/暂停检测逻辑被统一纳入新检测器，不出现重复分叉实现
